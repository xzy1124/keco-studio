### 1. `/specify`（规格定义）

- **核心作用**：明确 “要做什么”，把模糊的需求转化为结构化的规格文档。
- **具体动作**：通过 AI 辅助，用自然语言描述需求的 “用户场景、验收标准、边界条件”，最终生成 `spec.md`（功能规格文档）。
- **解决的问题**：避免 “需求理解偏差”，让团队 / AI 对 “功能目标” 达成共识。

#### 2. `/plan`（方案规划）

- **核心作用**：明确 “怎么做”，基于规格文档设计技术方案。
- **具体动作**：AI 分析规格，输出技术栈选型、架构设计、数据模型、API 规范等，生成 `plan.md`（技术方案文档）。
- **解决的问题**：避免 “技术选型混乱”“架构不合理”，确保实现路径可落地。

#### 3. `/tasks`（任务拆分）

- **核心作用**：把方案拆解为可执行的任务单元。
- **具体动作**：AI 按依赖关系拆分任务（比如 “先建数据库模型→再开发接口→最后写前端组件”），生成 `tasks.md`（带优先级、依赖的任务清单）。
- **解决的问题**：避免 “工作无序”，让开发过程按步骤推进，适合团队协作或个人高效执行。

#### 4. `/implement`（代码实现）

- **核心作用**：按任务生成代码，同时强制测试驱动开发（TDD）。
- **具体动作**：AI 按任务顺序生成代码，且必须先写测试用例（确保代码可验证），最终输出可运行的源代码 + 测试文件。
- **解决的问题**：避免 “代码质量低”“测试覆盖不足”，让实现结果既符合规格，又具备可维护性。

#### 补充：/constitution 的角色

流程图左侧的 `/constitution` 是 Spec-Kit 的 “项目宪法”（对应之前的 Constitution 文档），它是这四个命令的**约束基础**—— 所有 `/specify` `/plan` 等步骤都必须遵守 `/constitution` 定义的核心原则（比如 “测试优先”“库优先”），确保开发过程不偏离项目的顶层规则。



```plaintext
/spec-kit.constitution 代码编写原则
1. 设计还原：基于F2C MCP的Figma设计稿实现像素级还原，包含尺寸/样式/交互态，并且使用响应式布局；
2. 技术栈：
   - Next.js：遵循App Router，避免水合错误，组件按需导入；
   - Supabase：适配RLS策略，Migration脚本管理表结构，数据双端校验；
   - 样式：.module.css模块化，提取CSS变量，样式复用无全局污染；
3. 代码规范：
   - 最小依赖，锁定package.json版本，无冗余库；
   - TypeScript严格模式，无any类型，核心逻辑注释；
   - 异步操作全量错误处理，避免崩溃；
4. 简洁性：逻辑清晰，组件单一职责，命名语义化。
```
